To cover **similar cases properly**, I need data that lets us create **case summaries**, not raw sensor rows only.

Think of `similar_case_rag` like this:

```text
Current inference case
        ↓
Find previous cases with similar:
risk zone + TTE + uncertainty + confidence + alarms + subset + outcome
        ↓
Use those cases to justify recommendation
```

So I need files that contain **prediction outputs, uncertainty, actual TTE if available, risk zones, errors, and case identity**.

# 2. Exact Columns I Need Per Case

For each similar case, I need these categories.

***

## A. Case Identity Fields

These tell us **which engine and which point in time** the case belongs to.

Required if available:

```text
unit
sample_index
sample_role
subset
end_cycle
window_start_cycle
hidden_failure_cycle_est
source_file
source_row_id
```

### Why needed?

Because a case should not be anonymous.

Bad case:

```json
{
  "predicted_tte": 24.5,
  "risk_zone": "Critical"
}
```

Good case:

```json
{
  "case_id": "CASE_FD001_UNIT_014_CYCLE_185_CRITICAL",
  "subset": "FD001",
  "unit": 14,
  "sample_role": "critical_focus",
  "end_cycle": 185,
  "predicted_tte": 24.5,
  "risk_zone": "Critical"
}
```

This helps authenticity and retrieval.

***

## B. Prediction Fields

These tell us what the model predicted.

Needed:

```text
predicted_tte
boost_pred
cnn_pred
ensemble_pred
used_w_boost
used_w_cnn
conformal_q90
conformal_q95
```

### Why needed?

Because similar cases should explain:

```text
Was this prediction from the ensemble?
Did HGB and CNN agree?
Was one model much more optimistic?
What ensemble weights were used?
```

Example:

```json
{
  "boost_pred": 98.08,
  "cnn_pred": 111.57,
  "ensemble_pred": 103.07,
  "model_disagreement_abs": 13.49
}
```

This allows the recommendation system to retrieve cases where model disagreement was high.

***

## C. Uncertainty Fields

These are very important.

Needed:

```text
lower_bound_90
upper_bound_90
interval_width_90
lower_bound_95
upper_bound_95
interval_width_95
confidence
```

### Why needed?

Because similar cases should match not only point prediction, but also uncertainty.

For example:

```text
Case A:
predicted_tte = 105
lower_bound_95 = 98
confidence = High

Case B:
predicted_tte = 105
lower_bound_95 = 52
confidence = Low
```

Both have the same point prediction, but Case B is much riskier.

So similar-case retrieval must use:

```text
predicted_tte + lower_bound_95 + interval_width_95 + confidence
```

***

## D. Alarm and Risk Fields

Needed:

```text
risk_zone
predicted_zone
actual_zone
uq_risk_zone_90
uq_risk_zone_95
critical_alarm_30
critical_alarm_33
uq_alarm_90
uq_alarm_95
recommendation
uq_recommendation_90
uq_recommendation_95
```

### Why needed?

These fields help us classify cases into useful buckets:

```text
Critical
Warning
Caution
Safe
Safe but UQ Caution
Point Critical
UQ Critical
False Safe
Correct Safe
```

Example:

```json
{
  "risk_zone": "Safe",
  "uq_risk_zone_95": "Possible Caution",
  "uq_alarm_95": false,
  "recommendation": "Normal operation",
  "uq_recommendation_95": "UQ-aware caution: monitor closely"
}
```

That becomes a valuable case type:

```text
safe_point_but_uq_caution
```

***

## E. Actual Outcome / Evaluation Fields

If available, these are extremely useful:

```text
actual_tte
actual_tte_raw
actual_zone
residual
absolute_error
covered_90
covered_95
risk_zone_correct
```

### Why needed?

Without actual outcome, the case is useful but incomplete.

With actual outcome, we can know:

```text
Was the model correct?
Was the interval covering true TTE?
Was the risk zone correct?
Was the case a near miss?
Was it a high-error case?
```

Example:

```json
{
  "predicted_tte": 103.07,
  "actual_tte": 97.0,
  "absolute_error": 6.07,
  "covered_95": true,
  "risk_zone_correct": true
}
```

This becomes a trustworthy similar case.

***

## F. Explainability Fields

If available:

```text
hgb_shap_sample_top_features
hgb_shap_global_feature_importance
sample_shap_top_features
cnn_attention_top_timesteps
cnn_attention_concentration
chart_summary
```

### Why needed?

These fields let similar cases explain **why** the model behaved that way.

Example:

```json
{
  "top_features": [
    "s11_slope",
    "s7_late_minus_early",
    "cycle_norm_last"
  ],
  "attention_summary": {
    "top_timestep_position": "late_window",
    "attention_concentration": "high"
  }
}
```

Then the final recommendation can say:

```text
Similar cases with late-window degradation features and low TTE were handled conservatively.
```

# 4. Best Similar Case Types We Should Generate

Once you provide the files, I will generate these case groups.

***

## 1. Critical Cases

Condition:

```text
predicted_tte < 30
or actual_tte < 30
or lower_bound_95 < 30 / alarm threshold
```

Purpose:

```text
Teach the recommender what near-failure cases look like.
```

These are highest priority.

***

## 2. UQ-Alarm Cases

Condition:

```text
uq_alarm_95 = true
or uq_alarm_90 = true
```

Purpose:

```text
Teach the system when uncertainty should change the recommendation.
```

Very important for your system because UQ alarms reduced missed critical decisions in your metrics.

***

## 3. Safe Point but UQ Caution Cases

Condition:

```text
risk_zone = Safe
and uq_risk_zone_95 in ["Possible Caution", "Possible Warning", "Possible Critical"]
```

Purpose:

```text
Prevent overconfident normal-operation recommendations.
```

This is one of the most important case types for your RAG.

***

## 4. High-Uncertainty Cases

Condition:

```text
interval_width_95 is high
or confidence = Low
```

Purpose:

```text
Support conservative recommendations and expert review.
```

***

## 5. Top Error Cases

Condition:

```text
absolute_error is high
```

Purpose:

```text
Teach the system where model mistakes happen.
```

***

## 6. Not-Covered Cases

Condition:

```text
covered_95 = false
or covered_90 = false
```

Purpose:

```text
Teach the system uncertainty intervals are not perfect.
```

***

## 7. Model-Disagreement Cases

Condition:

```text
abs(boost_pred - cnn_pred) is high
```

Purpose:

```text
Support expert review or conservative scoring when HGB and CNN disagree.
```

***

## 8. Checkpoint-History Cases

Condition:

```text
sample_role = checkpoint_history
```

Purpose:

```text
Show how risk evolves over time for the same engine.
```

These are very useful for recommendation explanation.

***

## 9. Critical-Focus Cases

Condition:

```text
sample_role = critical_focus
```

Purpose:

```text
Stress-test behavior around the critical threshold.
```

***

## 10. Safe Confirmed Cases

Condition:

```text
risk_zone = Safe
uq_alarm_95 = false
covered_95 = true
actual_zone = Safe
```

Purpose:

```text
Prevent the system from becoming too conservative.
```
