# 1. Final Stack Decision

## Final DB

```text
Qdrant
```

Use **one Qdrant deployment/service**, not multiple DB servers.

Inside that one Qdrant instance, create **five separate collections**:

```text
domain_rag
similar_case_rag

policy_rag
failure_mode_rag
```

This is the best design for your system because each RAG branch has a different purpose, different metadata, different retrieval behavior, and different evidence-quality requirement. Qdrant supports vector storage with payload metadata and filtering, which fits this branch-wise RAG design. [\[analyticsvidhya.com\]](https://www.analyticsvidhya.com/blog/2025/09/embeddinggemma/)

***

## Final Embedder

```text
BAAI/bge-m3
```

Why BGE-M3:

```text
Dense retrieval       → semantic search
Sparse retrieval      → keyword / word-based search
Multi-vector retrieval → fine-grained matching
```

This is ideal for your use case because your input has both semantic summaries and exact technical terms like `TTE`, `PICP`, `MPIW`, `CRPS`, `SHAP`, `anomaly_score`, and `vibration_rms`. BGE-M3 is documented as supporting dense, sparse, and multi-vector retrieval, and it supports more than 100 languages with input length up to 8192 tokens. [\[betterstack.com\]](https://betterstack.com/community/guides/ai/gemma-4/), [\[brian.kitchen\]](https://www.brian.kitchen/blog/introduction-to-rag)

Key BGE-M3 specs:

```text
Model: BAAI/bge-m3
License: MIT
Embedding dimension: 1024
Max sequence length: 8192 tokens
Model size: ~569M parameters
Retrieval modes: dense + sparse + multi-vector
```

The model card lists BGE-M3 with 1024 dimensions, 8192 sequence length, and MIT licensing. [\[ollama.com\]](https://ollama.com/library/embeddinggemma), [\[prabs.xyz\]](https://prabs.xyz/blog/per-layer-embeddings-the-secret-powering-gemma-4-models)

***

## Final Retrieval Type

```text
Hybrid retrieval = dense semantic retrieval + sparse keyword retrieval
```

Dense retrieval helps with meaning-based queries.

Sparse retrieval helps with exact terms and metric names.

Qdrant supports hybrid search by combining dense and sparse vectors, with fusion methods like RRF and DBSF. [\[railway.com\]](https://railway.com/deploy/embedding-gemma), [\[huggingface.co\]](https://huggingface.co/Ccre/gemma-embedding-model)

***

## Final Reranker

Use one of these:

```text
Primary: BAAI/bge-reranker-v2-m3
Alternative: Qwen3-Reranker-0.6B
```

Your architecture already has:

```text
Context Merger + Reranker
```

So the retriever should fetch more candidates first, then the reranker selects the best evidence chunks. The BGE-M3 documentation also recommends hybrid retrieval followed by reranking for stronger RAG quality. [\[brian.kitchen\]](https://www.brian.kitchen/blog/introduction-to-rag), [\[prabs.xyz\]](https://prabs.xyz/blog/per-layer-embeddings-the-secret-powering-gemma-4-models)

# 4. Why Five Collections Instead of One?

You could start with one collection for MVP, but for your fixed architecture, five collections are better.

## Reason 1: Cleaner retrieval

Each branch has a different purpose:

```text
domain_rag        → general predictive-maintenance knowledge
similar_case_rag  → previous recommendation cases
metric_rag        → metric and UQ interpretation
policy_rag        → rules and constraints
failure_mode_rag  → fault/failure knowledge
```

Keeping them separate avoids retrieval pollution.

***

## Reason 2: Different retrieval behavior

Recommended retrieval style:

```text
domain_rag        → dense-heavy hybrid search
similar_case_rag  → dense + metadata filtering
metric_rag        → sparse-heavy hybrid search
policy_rag        → strict metadata + sparse-heavy search
failure_mode_rag  → balanced dense + sparse search
```

This is important because `metric_rag` and `policy_rag` need exact matching more than `domain_rag`.

***

## Reason 3: Better evidence quality checking

Your Evidence Quality Checker can verify coverage branch-wise:

```text
Did Domain RAG return useful evidence?
Did Similar Case RAG return a relevant case?
Did Metric RAG explain the important metrics?
Did Policy RAG return valid constraints?
Did Failure Mode RAG identify likely failure modes?
```

This becomes easier when each branch has its own collection.

# 6. What the Embedder Does and Does Not Do

Important point:

```text
BGE-M3 does not decide which collection the document goes into.
```

The collection is decided by:

```text
folder name
```

Then the embedder only does:

```text
text chunk → dense vector + sparse vector
```

Then your ingestion pipeline inserts that chunk into the matching Qdrant collection.

So the flow is:

```text
Document placed in folder
        |
        v
Folder name decides collection
        |
        v
Text is extracted and chunked
        |
        v
BGE-M3 creates dense + sparse embeddings
        |
        v
Chunk is inserted into matching Qdrant collection
```

# 8. Ingestion Pipeline

Your ingestion pipeline should work like this:

```text
1. Create Qdrant collections
2. Create matching folders
3. Place trusted documents manually in the correct folders
4. Read each folder
5. Extract text from documents
6. Clean text
7. Chunk text
8. Generate BGE-M3 dense + sparse embeddings
9. Insert chunks into corresponding Qdrant collection
10. Store metadata with every chunk
```

Detailed flow:

```text
rag_knowledge_base/domain_rag/docs/file.md
```

# 11. Query-Time Retrieval Flow

When your recommendation system receives structured JSON:

```json
{
  "predicted_tte": 36,
  "confidence": 0.78,
  "uncertainty_interval": {
    "lower": 24,
    "upper": 52
  },
  "risk_level": "high",
  "top_features": ["vibration_rms", "bearing_temperature"],
  "anomaly_score": 0.91,
  "chart_summaries": [
    "Vibration RMS increased continuously.",
    "Bearing temperature shows upward trend."
  ]
}
```

The Query Router creates five queries:

```text
Domain RAG query:
Interpret high-risk short TTE with rising vibration and temperature.

Similar Case RAG query:
Find previous cases with TTE 24–48 hours, high risk, vibration_rms, bearing_temperature.

Metric RAG query:
Explain TTE, uncertainty interval, confidence, anomaly score, SHAP/top features.

Policy RAG query:
Find policy for high risk and TTE below 48 hours.

Failure Mode RAG query:
Find likely failure modes for vibration increase, temperature rise, high anomaly score.
```

Each branch searches its own Qdrant collection using:

```text
BGE-M3 dense vector
+
BGE-M3 sparse vector
+
Qdrant hybrid search
```
